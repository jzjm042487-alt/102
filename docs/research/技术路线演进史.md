# 技术路线演进史：从 GPU 预解器到全局集合覆盖

> 本文记录蛇形管排料求解器"高利用率 + 少花样 + 全排完"目标下，技术路线从头到尾
> 经历的每一次尝试、遇到的问题、以及切换路线的**根本原因**。目的：让任何人（含未来的
> 我们自己）读完就能明白"为什么最终选全局集合覆盖，而不是别的"，避免走回头路。
>
> 详细数据与实验记录见 `GPU可插拔预解器设计方案.md`（本文是它的**演进主线摘要 + 决策链**）。
>
> 状态：路线已定（全局集合覆盖 / 列生成 ILP）。本文完成后即开始正式落盘该路线并做详细测试。

---

## 0. 问题是什么（一句话）

把**长管**（蛇形管，~22000mm）用**短母材**（6500~12400mm）切割 + 焊接拼出来，要求：

1. **全排完**：每根需求管都排出来；
2. **利用率高**：紧供料比（需求≈总料，1.00~1.02），任何废料都很致命；
3. **花样少**：切法种类、拼法种类都要少（现场按花样换刀/换工装，花样多=生产痛苦）；
4. **守红线**：`min_weld_distance`（焊口最小间距）、`max_joints`（最大焊口数）、
   禁焊区（`forbidden`/`Unweldable_Area`）、`min_cut_length`（最短可焊段）、kerf（切口损耗）。

对标物：用户现有**老排料软件**——花样优秀、能全排完，唯一缺点是**慢、常需人工手排（周期一周）**。
我们的验收线（业务拍板，§11.20）：**利用率 ≥ 老软件、焊口 ≤ 老软件、切法种类 ≤ 老软件、拼法种类 ≤ 老软件**——只要不劣化即可验收。

## 2. 每一次切换的详细原因

### ① GPU 加速候选生成 →（证伪）

- **出发点**：以为紧料组解不出是"候选列不够多/不够好"，想用 GPU 并行海量生成候选拼法列喂给 SCIP。
- **血泪教训（§一、§11.11）**：
  - "启发式暖启动"是错误的拐杖——用比 MILP 更弱的启发式 warm-start，会引入 MILP 本没有的可行性缺陷。
  - 真机 30 组 A/B（§11.1）：GPU 候选并未显著提高解出率。
  - 度量发现（§11.11）：**真瓶颈不在生成速度**，而在 MILP 求解本身；GPU 逐 multiset 调度比 CPU 慢**几千倍**，maximal-only 剪枝还会拖垮 MILP。
- **保留的扎实成果**：可插拔 accel 抽象接口、CPU no-op 退回、独立 verifier 一关不变（§11.3）。这些与"GPU 是否有价值"无关，全部留用。
- **结论**：GPU 不是当前矛盾主要方面，**下线**。

### ② 等价母材归约 CSP →（对小中组成立，大规模不足）

- **关键修正（§11.6）**：重新调研发现问题是**两阶段（切割+焊接）**，不是单阶段 CSP。Ilyés-Balogh (2026) 给出三步法：切割+焊接 → 等价标准 CSP → 解 CSP → 模式转换回切焊方案。
- **实测（§11.8/§11.9）**：模式转换（第③步）成功率 **100%**；自适应 alpha + 强制 min_weld_distance + 收紧词典序后，救活率明显提升。
- **问题**：超紧组仍 INFEASIBLE，等价母材 alpha 增大导致**组合爆炸**。
- **结论**：对小中组有效，作为**择优补强引擎**并入生产（→ 路线 C），但大规模需要另想办法。

### ③ 路线 A/C 生产加固 →（完成，但暴露 types-heavy 死角）

- **路线 A（§11.10）**：禁焊区、must_use 语义对齐生产红线，A/B 实测 enforce vs ignore 通过。
- **必用料语义修正（§11.16）**：从"按长度硬下界"改为**"按材质规格全局优先"**——必用料优先塞满，用尽仍不够才动可用料；需求小则只用够需求的必用料。
- **路线 C（§11.12/§11.14）**：route2 解 → 生产 schema → 真 `verify_solution` 通过；正式单测 18 用例全过；环境变量开关。
- **收口（§11.13/§11.15）**：超紧组 INFEASIBLE 根因量化；多 seed OFF vs ON 基准验收；修复末端自校验闸门。
- **暴露的死角**：**types-heavy（几百种段长）+ 千根级母材**的组，两层 MILP 和 route2 **都求不动**（组合爆炸），而老软件能解。这触发了 route3。

### ④ route3 顺序 DP →（兜底有效，但触顶）

- **思路（§11.17）**：MKA 钢铁业方案——不建全局模型，**逐管/逐母材顺序装配**，用小 DP 选每根母材切法，复杂度随母材根数**线性增长**。定位为大规模兜底路径。
- **实测**：702 段长 / baseline 真解不出的硬组，route3 把结果从 **0 管 → 1060/1104 管**，~2s。**兜底价值确认**。
- **但触顶**：利用率仅 ~96%，无法全排完。诊断根因：不是缺料，而是料被切成**小碎片**；44 根失败管都是 10991mm、`max_joints=3`，而剩料多是 1272mm 小段——拼一根要 ~9 段 8 个焊口，**远超 max_joints=3**。这是**全局资源分配问题**，局部逐管优化解不了。

### ⑤ route3 + 朴素 LNS →（触顶，用户叫停打补丁）

- **思路（§11.23）**：破坏-重建局部搜索——拆掉最浪费的母材、重建、词典序接受更优解。route3 已转成合格构造器（过独立 verifier，kerf 记账彻底修正为切口损耗模型，§11.22）。
- **实测**：朴素 LNS 能救活部分失败管、略降花样，但**贪心重建不够**，且"救活失败管"与"减少花样种类"在局部路线里**天然冲突**（词典序首位难拍）。
- **用户关键反馈（转折点）**：
  > "我看你的方案是逐步再去打补丁……应该做一个彻底的方案，大胆假设、小心求证……每次打补丁会导致后续根本无法执行。"
  >
  > "老算法都能算出切法优秀、也能全排完，是不是我们选择局部优化这条路线从一开始就错了？必须全局。"
- **结论**：局部优化路线**本质不匹配**"全排完 + 少花样"双目标。停止打补丁，转彻底方案。

### ⑥ 彻底方案调研 + 逆向老软件 →（钉死方向）

- **理论可解性（§11.18）**：问题 NP-hard；紧供料比 1.00~1.02 区正是**理论最难区**（大容量 + triplet 结构 + non-IRUP）——这解释了为什么纯精确法（MILP）会失败。**纯列生成/branch-and-price 也不宜做主引擎**（>700 段长时 tailing-off/退化）。
- **工业落地共识（§11.18/§11.19）**：真正做到"数千根 & 99%+"的都是**受限集合覆盖 matheuristic + 构造启发式（route3 血统）+ SA/LNS 局部搜索**的组合；pricing 子问题正确建模是**弧流图上的资源约束最短路（RCSP）**。深挖发现 SSP-DC（焊接侧精确对口）、arc-flow + graph compression、SVC、Filtered Beam Search 等可借鉴件（有现成开源代码）。
- **逆向老软件（§11.24，决定性）**：`scripts/_reverse_legacy_algo.py` 量化 4 笔最复杂样本的 6 个算法指纹：
  - **切法/母材比极低（0.019~0.13）、单切法复用高达 374 张** → 只有集合覆盖/列生成会有此签名，**GA/SA 被否定**。
  - **每管型几乎固定单一拼法**（同管多拼法 ≈ 0）→ 标准集合覆盖结构（管型=需求行，切法=列）。
  - **废料高度集中**（B 前 10% 母材吃 89% 废料）→ 全局精确把 trim 挤到少数"垃圾桶"母材。
- **结论**：**老软件主引擎 = 集合覆盖 / 列生成 ILP**。我方应对齐这条路，不再走局部优化。

### ⑦ 全局集合覆盖 ILP（POC）→（✅ 选定）

- **方法（§11.25，`scripts/_poc_setcover_ilp.py`）**：
  1. 每管型固定一种拼法模板（关节最少、过所有硬约束）→ 68 管型只落成 **17 种段长**；
  2. 秒级枚举各母材长度的切法列；
  3. 集合覆盖 ILP：`Σ 切法·根数 ≥ 段长需求`（→ **天然全排完**），最小化母材根数。
- **B 样本结果（一次跑通、全面优于老软件）**：

  | 指标 | POC | 老软件 |
  |---|---|---|
  | 切法种类 | **23** | 70 |
  | 拼法种类 | 68 | 68 |
  | 母材根数 | **545** | 546 |
  | 利用率 | **0.998874** | 0.997288 |
  | 耗时 | **1.19s** | 常需人工手排 |

- **关键工程教训**：显式数花样的 big-M `y_p` 变量是**唯一瓶颈**（加了它 SCIP 60s 无解，去掉后 1.2s 出最优）；而"最小化母材根数"在小段长字母表上**天然只复用极少切法**，无需显式数花样。换小列池配置还能压到 5 种切法 / 99.73%——"花样 vs 利用率"可在同一模型内调档。

## 3. 为什么最终是"全局集合覆盖"（决策逻辑闭环）

1. **目标决定路线**：要同时"全排完 + 少花样"，这两个目标在**局部贪心里天然冲突**（④⑤已实证），只有**全局模型**能一次性协调 → 排除所有局部路线。
2. **对标物指路**：老软件既优秀又全排完，逆向指纹证明它就是**集合覆盖/列生成 ILP**（⑥）→ 抄这条最稳，且我方已有 SCIP 依赖。
3. **文献背书**：NP-hard 最难区，纯精确法必败；工业界公认解法就是"受限集合覆盖 matheuristic + 构造启发式 + 局部搜索"（⑥）→ 与逆向结论完全吻合。
4. **POC 实证**：B 样本一次跑通、全面优于老软件、秒级（⑦）→ 方向钉死，可以投产工程。

## 5. 下一步（本文完成后即执行）

正式落盘全局集合覆盖路线 + 详细测试，建议顺序：

1. **规模升级验证**：把 POC 拿到 C 样本（5109 母材 / 44 管型，规模最大）与 A/D 重跑，确认千根级仍秒~分钟级可解。
2. **列生成兜底**：若静态全枚举列爆炸，引入 §11.19 的**列生成 / pricing（RCSP）**按需生列，而非全枚举。
3. **约束贯通**：把 must_use（材质规格全局优先）、禁焊区、min_weld/max_joints/min_cut/kerf 全部并入集合覆盖模型（前置④待补：must_use 在模型里的表达）。
4. **拼法花样收敛**：拼法固定单模板对齐老软件；必要时用少量备选模板做二次优化。
5. **过独立 verifier + 逐样本对标验收表**：全库 A/B 回归，确认对每笔样本都"不劣化"。
6. **并入生产 service**：作为独立子求解路径（复用路线 C 的接入与开关经验），保留退回。
